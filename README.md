# Football Forecasting Agent Harness

نموذج عملي لنظام توقع مباريات كرة القدم بمبدأ **Agent Harness**، مربوط حالياً بجدول كأس العالم 2026 من مصدر openfootball.

الفكرة ليست أنه نموذج مراهنات إنتاجي، بل أنه مسار توقع قابل للتشغيل والاختبار:

- وكيل تنسيق يشغل مساراً ثابتاً لكل مباراة.
- وكيل جمع بيانات يحمّل النتائج التاريخية وإشارات الأحداث المتأخرة.
- وكيل تحليل إحصائي يشغل نموذج Poisson مع تعديل Dixon-Coles مبسط.
- وكيل العوامل الفراشية يعطي الأحداث القريبة من موعد المباراة وزناً أعلى.
- وكيل نقد يعمل كـ Devil's Advocate ويبحث عن اعتراضات على التوقع.
- وكيل توليف يصدر الاحتمالات النهائية.
- وكيل مراجعة ذاتية يقارن التوقع بالنتيجة الفعلية عند توفرها.

https://raw.githubusercontent.com/openfootball/worldcup.json/master/2026/worldcup.json

## تحديث النتائج بعد المباريات

لتحديث ملف كأس العالم من المصدر المفتوح، ثم إعادة بناء ملفات البيانات والتوقعات:

```powershell
python .\update_worldcup_results.py
```

ولتشغيل وكيل مراقبة ينتظر زمن نهاية المباراة حسب توقيت `kickoff_utc` ثم يطلق التحديث تلقائياً:

```powershell
python .\match_monitor.py
```

أو تشغيله في الخلفية على ويندوز:

```powershell
.\start_match_monitor.ps1
```

وإيقافه:

```powershell
.\stop_match_monitor.ps1
```

السجل يظهر في:

```text
outputs/match_monitor.log
```

للتجربة مرة واحدة فقط بدون حلقة مستمرة:

```powershell
python .\match_monitor.py --once
```

المنطق الحالي ينتظر `130` دقيقة بعد بداية المباراة قبل أول فحص للنتيجة، ويمكن تغييره:

```powershell
python .\match_monitor.py --final-whistle-buffer 125
```

هذا السكربت ينفذ الدورة التطورية:

1. يسحب آخر النتائج من المصدر المفتوح.
2. يعيد بناء ملفات النتائج والمباريات المتبقية.
3. يقارن التوقعات القديمة بالنتائج الفعلية التي ظهرت.
4. يحسب دقة كل مباراة وملخص الدقة الكلي.
5. يحفظ نسخة كاملة من النموذج والتوقعات داخل:

```text
outputs/model_versions/
```

6. يحدّث `data/model_calibration.json` بشكل محافظ.
7. يعيد توقع بقية مباريات المجموعات ومسارات دور الـ32.

الواجهة المحلية تعيد قراءة ملفات التوقع كل دقيقة ما دامت الصفحة مفتوحة، لكن يجب أن يعمل سكربت التحديث أعلاه حتى تصل النتائج الجديدة من المصدر إلى ملفات المشروع.

## دقة النموذج

بعد تشغيل التحديث التطوري يظهر:

```text
outputs/accuracy_report.json
```

ويحتوي:

- دقة كل مباراة مكتملة.
- Brier Score لكل مباراة.
- Log Loss لكل مباراة.
- Accuracy الكلية.
- متوسط Brier وLog Loss.
- رابط نسخة النموذج التي صدر منها التقييم.

الداشبورد يحتوي تبويب `دقة النموذج` لعرض هذه النتائج.

## إحصاءات FIFA/beIN المتقدمة

يوجد ملف إدخال اختياري:

```text
data/match_advanced_stats.csv
```

الأعمدة المدعومة:

```text
match_id, team, shots, shots_on_target, possession, xg, big_chances, corners,
yellow_cards, red_cards, passes_into_final_third
```

عند إضافة أرقام مباراة من FIFA Match Centre أو beIN أو أي مصدر مفتوح موثوق، سيقرأها `TeamIntelligenceAgent` ويدخلها في حساب الترجيح. إذا لم توجد هذه الأرقام، يظهر تحذير جودة بيانات داخل بطاقة المباراة.

في الداشبورد: اضغط على بطاقة أي مباراة لفتح لوحة تفاصيل تعرض:

- التوقع قبل المباراة.
- النتيجة الفعلية ودقة النموذج بعد التحديث.
- Brier Score وLog Loss للمباراة.
- رسوم مقارنة للإحصاءات: xG، التسديدات، التسديدات على المرمى، الاستحواذ، الركنيات.
- العوامل التي رجحت التوقع.

## الملفات المهمة

- `football_harness/core.py`: تعريف `AgentHarness` وحالة المباراة وسجل التدقيق.
- `football_harness/model.py`: نموذج Dixon-Coles Lite.
- `football_harness/agents.py`: الوكلاء المنفصلون.
- `data/historical_results.csv`: مباريات كأس العالم المكتملة من المصدر.
- `data/fixtures.csv`: مباريات كأس العالم المطلوب توقعها.
- `data/team_seed_ranks.csv`: seed priors لقوة المنتخبات، تستخدم عندما يكون عدد مباريات المنتخب في البطولة قليلاً.
- `data/butterfly_events.csv`: أحداث عالية الأثر مع دالة وزن زمني.
- `data/actual_results.csv`: نتائج فعلية للمراجعة الذاتية.
- `data/team_visuals.csv`: أسماء المنتخبات بالعربية وألوانها في الداشبورد.
- `update_worldcup_results.py`: تحديث المصدر المفتوح ثم إعادة توليد التوقعات.

## ملاحظة دقة

النموذج يعرض الآن عوامل الترجيح داخل كل بطاقة: أداء البطولة، فارق الأهداف، seed prior، إشارات أثر الفراشة، وتحذيرات نقص البيانات. إحصائيات FIFA/beIN المتقدمة مثل التسديدات، xG، الاستحواذ، PPDA، والتمريرات التقدمية ليست مربوطة بعد كمصدر آلي، لذلك تظهر كتحذير جودة بيانات إلى أن تُضاف كملف إدخال أو API.

## التوسعة التالية

الخطوة التالية الطبيعية هي استبدال بيانات العينة بمصدر تاريخي حقيقي، ثم تحويل كل وكيل إلى function-calling contract بحيث يمكن ربطه لاحقاً مع Claude API أو أي LLM آخر دون تغيير جوهر الـ Harness.
